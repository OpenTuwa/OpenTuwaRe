# SEO Infrastructure — Design

## Overview

OpenTuwa is a Next.js (App Router, edge runtime) news site deployed on Cloudflare Pages with Cloudflare Workers serving the `functions/` directory. The site uses a dual-rendering strategy: real users receive the Next.js SPA; bots receive a server-rendered HTML shell generated by Cloudflare Worker functions.

This design covers four pillars:

1. **Technical SEO & Crawl Efficiency** — robots.txt, sitemap hierarchy, canonical tags, HTTP headers
2. **Enterprise News Entity Schema** — JSON-LD `@graph` with `NewsMediaOrganization`, `NewsArticle`, `WebSite`, `BreadcrumbList`
3. **DOM Semantics & WCAG 2.2 AAA Accessibility** — landmark roles, heading hierarchy, ARIA labels, focus management, colour contrast
4. **Core Web Vitals Performance** — LCP, INP, CLS targets via resource hints, image optimisation, layout stability

The design targets Google News eligibility, rich-result eligibility (Article, Breadcrumb, Sitelinks Searchbox), and Lighthouse scores ≥ 90 across all four categories.

### Dual-Rendering Strategy

The existing `isBot()` gate in every Worker function is the correct pattern. Bots receive a fully-rendered HTML response with all meta tags and JSON-LD inline. Real users receive the Next.js SPA which hydrates structured data via React components. Both paths must emit identical canonical URLs and equivalent JSON-LD.

## Components and Interfaces

### 1. `functions/robots.txt.js`

Current state is functional. Gaps:
- Missing `Crawl-delay` directive for aggressive bots
- `Disallow: /_next/static/` should be explicit (currently `/_next/`)
- Should reference `sitemap-index.xml` as the single entry point, not all three sitemaps

**Interface:** `onRequestGet(context) → Response<text/plain>`

### 2. Sitemap Hierarchy

```
/sitemap-index.xml   ← entry point for all crawlers
  /sitemap.xml       ← static pages + all articles (image sitemap extension)
  /news-sitemap.xml  ← articles published within last 48 h (Google News)
```

`sitemap.xml` must include `<lastmod>` derived from `updated_at` (not `NOW()`), `<priority>` and `<changefreq>` per page type, and `<image:image>` blocks for articles with images.

`news-sitemap.xml` must include `<news:genre>` mapped from `article.section`, `<news:keywords>` from tags (≤ 10), and `<news:language>en</news:language>`.

### 3. `StructuredData.jsx` — `@graph` consolidation

Current state emits multiple separate `<script type="application/ld+json">` blocks. Google recommends a single `@graph` array per page for entity disambiguation.

**New interface:**

```js
// Single component, replaces all individual schema components
<GraphSchema type="article" data={article} author={author} />
<GraphSchema type="homepage" />
<GraphSchema type="archive" />
```

Internally builds one `@graph` array:

- Homepage: `[WebSite, NewsMediaOrganization, BreadcrumbList]`
- Article: `[NewsArticle, Person (author), NewsMediaOrganization, BreadcrumbList, VideoObject?]`
- Archive: `[CollectionPage, NewsMediaOrganization, BreadcrumbList]`

### 4. `SEOHead.jsx`

Consolidate per-page components into a single `<PageSEO>` component accepting a `page` prop (`'home' | 'article' | 'archive' | 'about' | 'legal'`). This eliminates the five near-identical components.

### 5. Bot SSR HTML (`functions/index.js`, `functions/articles/[slug].js`)

Both files must emit the same `@graph` JSON-LD as the React layer. Extract a shared `buildGraphSchema(type, data)` utility in `functions/_utils/schema.js`.

### 6. Accessibility — `ArticleView.jsx` and layout components

Key changes:
- `<article>` element already present; ensure `aria-labelledby` points to the `<h1>` id
- `<nav>` elements need unique `aria-label` values (currently some are duplicated)
- Reading progress bar needs `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- Search input needs `role="search"` on its wrapper and `aria-label`
- Skip-to-content link must be the first focusable element in the DOM
- Colour contrast: `--tuwa-muted` (`#a1a1aa`) on `#0a0a0b` background = 5.74:1 (passes AA, fails AAA 7:1). Must be raised to `#c4c4c4` or equivalent for AAA body text

### 7. Core Web Vitals

**LCP** — hero image in `ArticleView.jsx` must have `fetchpriority="high"` and `loading="eager"`. `<link rel="preload">` for the hero image should be injected in the `<head>` server-side.

**INP** — the scroll handler in `ArticleView.jsx` already uses `requestAnimationFrame` throttling. The `track-interaction` fetch calls should be wrapped in `requestIdleCallback` to avoid blocking input.

**CLS** — `SkeletonImage` component must reserve explicit `width`/`height` or use `aspect-ratio` CSS to prevent layout shift. Article hero image must have explicit dimensions.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: robots.txt disallows private paths

*For any* robots.txt response generated by the system, the parsed directives must include `Disallow: /api/` and `Disallow: /_next/` for the `User-agent: *` group, and must not disallow `/`, `/articles/`, `/archive`, or `/about`.

**Validates: Requirements 1.1**

---

### Property 2: Sitemap article coverage

*For any* set of articles stored in the database, every article must appear in `/sitemap.xml` with a valid `<loc>` and `<lastmod>`, and every article that has a non-empty `image_url` must have a corresponding `<image:image>` block.

**Validates: Requirements 1.3**

---

### Property 3: News sitemap recency filter

*For any* set of articles, `/news-sitemap.xml` must contain only articles whose `published_at` is within the last 48 hours, and every included article must have a `<news:news>` block with `<news:publication_date>` and `<news:title>`.

**Validates: Requirements 1.4**

---

### Property 4: Single canonical URL per page

*For any* page rendered by the system (both bot SSR and Next.js metadata), the output must contain exactly one canonical link/tag, and its `href` value must not end with a trailing slash (except for the bare root `/`).

**Validates: Requirements 1.5**

---

### Property 5: Variant pages are noindex

*For any* request URL that contains a `?author=`, `?tag=`, or `?q=` query parameter, the rendered `<meta name="robots">` content must include `noindex`.

**Validates: Requirements 1.6**

---

### Property 6: Article @graph completeness

*For any* article rendered by the system, the emitted JSON-LD must be a single `<script type="application/ld+json">` block whose root object contains a `@graph` array with at least one node of each type: `NewsArticle`, `Person` or `Organization` (author), `NewsMediaOrganization`, and `BreadcrumbList`.

**Validates: Requirements 2.1**

---

### Property 7: dateModified never precedes datePublished

*For any* article schema object produced by `buildArticleGraph` (or the React equivalent), if both `datePublished` and `dateModified` are present, then `dateModified >= datePublished` must hold as ISO-8601 string comparison.

**Validates: Requirements 2.2**

---

### Property 8: Tags round-trip to keywords string

*For any* tags value stored in the database (whether a JSON array string like `'["a","b"]'` or a CSV string like `"a, b"`), the schema builder must produce a non-empty, comma-separated `keywords` string containing all the original tag values.

**Validates: Requirements 2.4**

---

### Property 9: Bot SSR and React schema parity

*For any* article, the JSON-LD `@graph` emitted by the Cloudflare Worker bot-SSR path and the JSON-LD emitted by the React `GraphSchema` component must contain the same set of `@type` values and the same `headline`, `datePublished`, `dateModified`, and `mainEntityOfPage` field values.

**Validates: Requirements 2.6**

---

### Property 10: Single h1 per page

*For any* page rendered by the system, the DOM must contain exactly one `<h1>` element.

**Validates: Requirements 3.2**

---

### Property 11: Unique aria-labels on nav elements

*For any* page that contains more than one `<nav>` element, all `aria-label` attribute values across those nav elements must be distinct (no two nav elements share the same label).

**Validates: Requirements 3.4**

---

### Property 12: Body text contrast meets WCAG AAA

*For any* foreground/background colour token pair used for body text in the design system, the WCAG 2.1 relative luminance contrast ratio must be ≥ 7.0:1.

**Validates: Requirements 3.5**

---

### Property 13: Hero image has priority loading attributes

*For any* article page where `image_url` is non-empty, the rendered hero `<img>` element must have `fetchpriority="high"` and `loading="eager"` attributes.

**Validates: Requirements 4.1**

---

### Property 14: SkeletonImage reserves layout space

*For any* rendered `SkeletonImage` component, the root element must have either explicit `width` and `height` attributes or a CSS `aspect-ratio` property set, so that the browser can reserve space before the image loads.

**Validates: Requirements 4.2**

---

### Property 15: Static assets have immutable cache headers

*For any* response for a path matching `/assets/*` or `/img/*`, the `Cache-Control` header must contain `immutable` and `max-age=31536000`.

**Validates: Requirements 4.4**

## Error Handling

### Schema builder errors

- If `article.published_at` is missing or unparseable, `datePublished` must be omitted rather than emitting `null` or `"Invalid Date"` — an invalid date in JSON-LD causes Google to reject the entire schema block.
- If `image_url` is a relative path, it must be resolved to an absolute URL using the site origin before inclusion in any schema or meta tag.
- If `tags` cannot be parsed as JSON and cannot be split as CSV, `keywords` must be omitted rather than emitting an empty string.

### Sitemap errors

- If the D1 database query fails, the sitemap Worker must return a valid (but empty) XML document with a 200 status rather than a 500, to avoid Googlebot treating the sitemap as broken.
- `<lastmod>` values must be validated as ISO-8601 before inclusion; invalid dates must be omitted.

### Bot SSR errors

- If the article is not found in the DB, the Worker must call `context.next()` to fall through to the Next.js SPA rather than returning a 404 directly, preserving the SPA's own 404 handling.
- All user-supplied strings interpolated into HTML must pass through the `esc()` function to prevent XSS in bot-rendered pages.

### Canonical URL errors

- Canonical URLs must always use `https://` and the production domain `opentuwa.com`, never the request origin, to prevent staging/preview deployments from emitting incorrect canonicals.

## Testing Strategy

### Dual Testing Approach

Both unit tests and property-based tests are required. Unit tests cover specific examples, integration points, and edge cases. Property tests verify universal correctness across randomly generated inputs.

### Property-Based Testing

**Library:** [fast-check](https://github.com/dubzzz/fast-check) (JavaScript/TypeScript, works in Node and edge environments)

Each property test must run a minimum of **100 iterations**. Each test must include a comment tag in the format:

```
// Feature: seo-infrastructure, Property N: <property_text>
```

**Property test targets:**

| Property | Test file | fast-check arbitraries |
|---|---|---|
| P2: Sitemap article coverage | `sitemap.test.js` | `fc.array(fc.record({slug, title, image_url, published_at}))` |
| P3: News sitemap recency | `news-sitemap.test.js` | `fc.array(fc.record({...}))` with random `published_at` offsets |
| P4: Single canonical | `canonical.test.js` | `fc.record({slug, params})` |
| P5: Variant noindex | `robots-meta.test.js` | `fc.oneof(fc.string(), fc.constant(''))` for each param |
| P6: @graph completeness | `schema.test.js` | `fc.record({article fields})` |
| P7: dateModified guard | `schema.test.js` | `fc.tuple(fc.date(), fc.date())` |
| P8: Tags round-trip | `schema.test.js` | `fc.oneof(fc.array(fc.string()), fc.string())` |
| P9: Bot/React parity | `schema-parity.test.js` | `fc.record({article fields})` |
| P10: Single h1 | `dom.test.js` | `fc.record({article fields})` rendered via jsdom |
| P11: Unique nav labels | `dom.test.js` | rendered page DOM |
| P12: Contrast ratio | `contrast.test.js` | `fc.tuple(fc.hexaString(6,6), fc.hexaString(6,6))` from design tokens |
| P13: Hero image attrs | `article-image.test.js` | `fc.record({image_url: fc.webUrl()})` |
| P14: SkeletonImage space | `skeleton-image.test.js` | `fc.record({src, width, height})` |
| P15: Cache headers | `headers.test.js` | `fc.constantFrom('/assets/logo.png', '/img/photo.jpg', ...)` |

### Unit Tests

Unit tests focus on:

- **Edge cases:** article with no image, article with no tags, article with `updated_at < published_at`, empty sitemap (no articles in DB)
- **Integration:** Worker function returns correct `Content-Type` headers; `sitemap-index.xml` references correct child URLs
- **Examples:** homepage `@graph` contains `WebSite` with `potentialAction`; skip-to-content link is first focusable element; reading progress bar has `role="progressbar"`

**Test runner:** Vitest (compatible with edge runtime via `@cloudflare/vitest-pool-workers` for Worker tests)

### Accessibility Testing

- Automated: `axe-core` integrated into Vitest via `@axe-core/react` for component-level checks
- Manual: keyboard navigation audit, screen reader test with NVDA/VoiceOver for article page flow
- Contrast: computed programmatically in unit tests using the WCAG relative luminance formula

### Performance Testing

- Lighthouse CI in GitHub Actions on every PR, asserting scores ≥ 90 for Performance, Accessibility, Best Practices, SEO
- CLS budget: < 0.1 measured via `web-vitals` library in a Playwright smoke test
- LCP budget: < 2.5 s on a simulated 4G connection
